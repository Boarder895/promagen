Batch Runner

Runs Call 3 → Score for each platform × scene. The core regression testing tool. 40 platforms × 8 scenes = 320 results per full batch.

Single platform diagnostic (fast, ~30 seconds)
npx tsx scripts/builder-quality-run.ts --platform lexica --mode builder 📋
Tests one platform across 8 scenes. Good for quick checks after a builder edit.
Full batch (all 40 platforms, ~15–20 minutes)
npx tsx scripts/builder-quality-run.ts --all --mode builder 📋
Decision-grade (3 replicates — use before shipping a builder change)
npx tsx scripts/builder-quality-run.ts --all --mode builder --replicates 3 📋
Include holdout scenes (release candidates only)
npx tsx scripts/builder-quality-run.ts --all --mode builder --holdout 📋
Compare against a baseline run
npx tsx scripts/builder-quality-run.ts --all --mode builder --baseline bqr-REPLACE-WITH-RUN-ID 📋
Replace bqr-REPLACE-WITH-RUN-ID with an actual run ID from a previous batch.
02

Rerun & Resume

Recovery mechanics. Rerun = re-run only failed results (creates a child run). Resume = continue a crashed run in-place.

Rerun failed results from a prior run
npx tsx scripts/builder-quality-run.ts --rerun bqr-REPLACE-WITH-RUN-ID 📋
Creates a new child run linked to the original. Only re-runs error results.
Resume a crashed/interrupted run
npx tsx scripts/builder-quality-run.ts --resume bqr-REPLACE-WITH-RUN-ID --force 📋
Continues the same run. Skips completed results, retries errors in-place. --force required for stale runs (no progress for 10+ min).
Ctrl+C is safe. The SIGINT handler marks the run as 'partial' and logs which platform/scene was last being processed. Resume picks up exactly where it left off.
03

User Sampling

Scores real user prompts from Community Pulse. Answers: "how good were the prompts we actually delivered?" Each entry is scored once (idempotency enforced).

Dry run (see what would be sampled, no scoring)
npx tsx scripts/builder-quality-sample.ts --dry-run 📋
Full sample (all platforms with user entries)
npx tsx scripts/builder-quality-sample.ts 📋
Single platform sample
npx tsx scripts/builder-quality-sample.ts --platform midjourney 📋
Custom look-back window
npx tsx scripts/builder-quality-sample.ts --days 14 --max-per-platform 3 📋
Idempotency: Running the sampler twice on the same window scores zero new entries. Already-scored entries are excluded automatically.
04

Failure Analysis

Clusters recurring failures for a platform. Nominates a fix class + target section. Requires 50+ results and 3+ scorer versions.

Analyse one platform
npx tsx scripts/builder-quality-analyse.ts --platform bing 📋
Outputs a structured failure report + fix-class nomination. Saves to reports/failure-report-bing.json.
Find the worst platforms across all
npx tsx scripts/builder-quality-analyse.ts --all --top 5 📋
Lower the data threshold (if you don't have 50 results yet)
npx tsx scripts/builder-quality-analyse.ts --platform bing --min-results 20 --scorer-versions 1 📋

Patch Suggestion

Uses GPT to write ONE targeted edit for the nominated fix class. GPT does not diagnose — it only writes the rule for the section chosen by the analysis.

Preview the GPT prompt (no API call)
npx tsx scripts/builder-quality-suggest.ts --platform bing --dry-run 📋
Generate a suggestion
npx tsx scripts/builder-quality-suggest.ts --platform bing 📋
Auto-runs analysis first if no failure report exists. Saves to reports/patch-suggestion-bing.json.
06

Patch Testing

Tests a patched builder through the actual /api/optimise-prompt route (full compliance + post-processing). Compares core + holdout against baseline. Applies overfitting guard.

1
Copy the builder file

Create a .patched copy to edit without touching the deployed file.

2
Apply the suggested rule

Open the .patched file, add the suggested rule to the nominated section.

3
Run the patch test

Tests through actual route with system prompt override. 3 replicates × 10 scenes.

4
Review verdict

SAFE = apply. CAUTION = review regression. REJECT = discard.

5
If SAFE: replace the builder file, run a normal batch to confirm

Final verification through the standard batch runner.

Step 1 — Copy builder
copy src\lib\optimise-prompts\group-nl-bing.ts src\lib\optimise-prompts\group-nl-bing.ts.patched 📋
Step 3 — Test the patch
npx tsx scripts/builder-quality-test-patch.ts --platform bing --patched-file src\lib\optimise-prompts\group-nl-bing.ts.patched 📋
Step 5 — After applying, run a normal batch to confirm
npx tsx scripts/builder-quality-run.ts --platform bing --mode builder --replicates 3 📋

Database Queries

Run these in the Neon SQL Editor (neon.tech → your project → SQL Editor).

Check worst scores from a run
SELECT platform_id, scene_id, gpt_score, anchors_preserved, anchors_dropped FROM builder_quality_results WHERE run_id = 'REPLACE' ORDER BY gpt_score ASC LIMIT 10; 📋
Check for duplicate rows (should return zero)
SELECT run_id, platform_id, scene_id, replicate_index, COUNT(*) FROM builder_quality_results GROUP BY run_id, platform_id, scene_id, replicate_index HAVING COUNT(*) > 1; 📋
View recent runs
SELECT run_id, mode, scope, status, mean_gpt_score, total_completed, total_expected, created_at FROM builder_quality_runs ORDER BY created_at DESC LIMIT 10; 📋
User sample rolling average (7 days)
SELECT platform_id, AVG(gpt_score)::numeric(5,2) AS mean, COUNT(*) AS n FROM builder_quality_results WHERE source = 'user_sample' AND created_at > NOW() - INTERVAL '7 days' AND status = 'complete' GROUP BY platform_id ORDER BY mean ASC; 📋
Clean up a failed run (replace ID)
DELETE FROM builder_quality_results WHERE run_id = 'REPLACE'; DELETE FROM builder_quality_runs WHERE run_id = 'REPLACE'; 📋

Troubleshooting

"BUILDER_QUALITY_KEY not set"

Add BUILDER_QUALITY_KEY=your-secret-key to .env.local. Any string works — it just needs to match between the script and the route.

"column bqr.showcase_entry_id does not exist"

The Part 11 schema hasn't been created yet. Run a batch (--platform lexica --mode builder) first — its ensureTables() creates the columns.

"OPENAI_API_KEY not set"

The validation harness and suggest script need OPENAI_API_KEY in .env.local. The batch runner uses it via the dev server route (which reads it from env).

502 errors during batch runs

Usually Vercel/dev server timeout. The runner retries 502s on scoring (falls back to scoring without anchor audit). If persistent, check the dev server terminal for errors.

Run stuck as 'running' after crash

Use --resume <run_id> --force to pick up where it left off. The SIGINT handler prevents this in most Ctrl+C cases.
